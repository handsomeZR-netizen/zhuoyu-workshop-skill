# 使用指南

琢玉工坊适合在你已经有一些项目材料时使用。材料可以很粗糙，但必须真实存在，例如代码仓库、报告草稿、截图、实验表格、老师要求、竞赛通知、README、报错日志。

## 推荐流程

1. 先做项目诊断，不急着写 PPT 或报告。
2. 再确定最终用途：课程汇报、大创结题、竞赛初赛、毕设答辩、GitHub 展示。
3. 补齐证据：截图、测试记录、数据来源、部署地址、演示视频、提交记录。
4. 生成交付物：PPT 大纲、讲稿、报告章节、README、材料清单。
5. 用验收清单检查：事实是否可验证，表达是否过度包装，格式是否符合要求。

```text
使用琢玉工坊帮我处理这个学生项目。

项目名称：
项目类型：课程设计 / 大创 / 竞赛 / 毕设 / 科研小项目
技术栈：
已有材料：
目标用途：
目标受众：
希望输出：
限制条件：
不能虚构的内容：
```

## 高质量输入模板

```text
项目名称：AI 电学实验辅导系统
项目类型：大创结题
技术栈：Next.js、Supabase、Python、OpenAI API
已有材料：代码仓库、README、3 张系统截图、结题报告初稿、一次演示视频
目标用途：校级大创结题答辩
目标受众：学院评委老师，偏工程和教学应用
希望输出：项目诊断、成果矩阵、10 页 PPT 大纲、5 分钟讲稿、评委 Q&A
限制条件：不能虚构上线用户量；没有论文和软著；模型效果没有正式实验数据
```

## 输出怎么验收

合格输出应该满足：

- 每个亮点都有证据来源。
- PPT 和报告结构能对应项目真实材料。
- 没有“国际领先”“行业首创”等无法证明的表述。
- 能清楚解释项目做了什么、怎么做、效果如何、还有什么不足。
- 如果涉及代码修改，必须有验证命令和失败回滚方式。

## 常见误区

- 只有技术名词，没有项目场景。
- 只说“系统实现了功能”，没有截图、日志或测试证据。
- 把未完成的功能写成已经完成。
- 答辩 PPT 页数太多，但每页没有明确结论。
- README 只写安装命令，不写项目价值、功能边界和演示方式。
