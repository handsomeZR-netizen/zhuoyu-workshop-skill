# 场景配方

## 大创或竞赛项目包装

适用场景：项目已经做出原型，需要整理成申报、结题或初赛材料。

推荐让 Codex 输出：

- 项目一句话定位。
- 真实用户场景和痛点。
- 技术路线图文字版。
- 成果矩阵：功能、证据、状态、风险。
- 结题报告或申报书章节结构。
- 评委可能追问的问题。

关键限制：不要虚构用户量、应用单位、论文、专利、软著、获奖情况。

## 课程设计或毕设答辩

适用场景：有代码和报告，需要快速形成答辩材料。

推荐让 Codex 输出：

- 8 到 12 页 PPT 大纲。
- 每页标题、核心结论、图示建议和讲稿。
- 3 到 5 分钟演示路线。
- 技术难点和解决方案。
- 不足与改进方向。

关键限制：不要把课程项目讲成商业落地产品，重点放在工程实现和学习成果。

## GitHub 项目展示

适用场景：需要让仓库看起来专业、可信、可运行。

推荐让 Codex 输出：

- README 新结构。
- 项目截图和演示 GIF 的放置建议。
- 快速开始命令。
- 功能列表和技术架构。
- 已知限制、路线图、贡献说明。

关键限制：不要写不存在的在线 Demo、测试覆盖率、Star 数或用户反馈。

## 论文和课程报告

适用场景：报告初稿结构混乱，摘要、章节、参考文献或格式不规范。

推荐让 Codex 输出：

- 摘要优化版本。
- 章节结构重排建议。
- 图表编号和引用检查清单。
- 术语统一表。
- 需要补证据的结论列表。

关键限制：不要修改实验数据结论，不要编造参考文献。

## Codex 编程辅助

适用场景：需要修报错、加小功能、做代码审查或整理项目结构。

推荐让 Codex 输出：

- 当前问题判断。
- 最小修改方案。
- 允许修改和禁止修改的文件范围。
- 验证命令。
- 回滚方式。

关键限制：先分析再修改，不要大范围重构，不要覆盖用户未提交修改。
