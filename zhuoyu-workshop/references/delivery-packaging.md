# 成果包装与交付材料

Use this reference when producing closing reports, achievement matrices, project portfolios, GitHub README drafts, software copyright material drafts, or competition submission copy.

## Delivery Package Structure

For student projects, organize materials into:

- **项目概述**: background, problem, objective, target users.
- **技术实现**: architecture, modules, data flow, key algorithms, implementation details.
- **成果展示**: functions, screenshots, demo route, experiment or test records.
- **项目管理**: division of labor, milestones, iteration process, risk handling.
- **总结展望**: limitations, next steps, realistic application scenarios.

## Achievement Matrix

Always prefer a matrix over vague narrative:

```markdown
| 成果项 | 类型 | 当前状态 | 证据材料 | 可放入哪里 | 风险 |
|---|---|---|---|---|---|
| 用户登录与权限 | 功能 | 已完成 | 截图、代码路径 | PPT/README/报告 | 需说明测试账号 |
```

Types can include: feature, document, experiment, dataset, model, demo, code, deployment, video, paper draft, software copyright draft.

## GitHub README Structure

For a student project README, include:

1. Project name and one-sentence value.
2. Screenshot or demo placeholder.
3. Feature list with completion status if needed.
4. Tech stack and architecture.
5. Quick start.
6. Environment variables and configuration.
7. Project structure.
8. Demo workflow.
9. Known limitations.
10. Roadmap.

Do not include fake badges, fake stars, fake coverage, fake online demo links, or fake production users.

## Closing Report Structure

Use this structure when no school template is provided:

1. 项目背景与意义。
2. 研究或开发目标。
3. 技术路线与系统设计。
4. 主要实现内容。
5. 测试、实验或演示结果。
6. 项目成果与创新点。
7. 经费或过程管理, if required.
8. 问题、不足与改进。
9. 总结。

## Software Copyright Draft Boundary

If the user asks for software copyright material, help draft descriptions and module lists, but do not claim the software copyright has been granted. Use wording such as:

- "用于软著申请材料准备的功能说明草稿"
- "建议由团队根据实际代码和学校/代理机构要求进一步核对"
