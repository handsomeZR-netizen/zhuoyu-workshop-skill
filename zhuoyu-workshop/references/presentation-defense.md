# 答辩与展示

Use this reference for PPT outlines, speaker notes, demo scripts, defense rehearsals, and judge Q&A.

## Defense Narrative

Use a simple arc:

1. Why this problem matters.
2. What the project built.
3. How the system works.
4. What evidence proves it works.
5. What is valuable, limited, and next.

For student projects, clarity beats novelty. Judges usually care whether the team actually built it, understands it, and can explain tradeoffs.

## PPT Page Formula

Each page should have:

- Page title as a conclusion, not a vague label.
- One main point.
- One recommended visual: screenshot, architecture diagram, flowchart, table, timeline, or demo frame.
- Speaker note for 20 to 40 seconds.
- Evidence source.

## Common 8-Page Structure

1. 项目背景与目标。
2. 用户场景与需求。
3. 系统整体方案。
4. 核心功能展示。
5. 技术实现与架构。
6. 测试、实验或演示结果。
7. 创新点与成果。
8. 不足、展望与总结。

## Demo Script

Keep the demo short and deterministic:

```markdown
## 演示路线
1. 打开系统首页，说明目标用户。
2. 使用测试账号登录。
3. 展示核心功能 A。
4. 展示核心功能 B。
5. 展示结果记录或后台数据。
6. 回到总结页，说明当前限制。
```

Always include fallback screenshots in case live demo fails.

## Judge Q&A Categories

Prepare questions for:

- User need: why this problem is real.
- Technical route: why this stack or model was chosen.
- Data and evaluation: what data was used and how results were checked.
- Team contribution: who did what.
- Innovation: what is different from existing systems.
- Limitation: what is not solved yet.
- Reproducibility: whether the project can run on another machine.
